Using Virtual Attributes

Active Roles allows you to extend the Active Roles Schema so that additional properties can be handled for existing object classes. Those properties are stored in the virtual attributes you can create with the Add Virtual Attribute Wizard provided by the Active Roles console.

This topic describes how to create virtual attributes and how to manipulate with their values.

Types of Virtual Attributes

Active Roles supports the following types of virtual attributes (VA):

  • Built-in virtual attributes: Built-in VAs are created at the Active Roles Administration Service installation time. These attributes are only calculated when used and are not stored in the Active Roles configuration database. The objects that represent the built-in VAs (objects of the attributeSchema type ) are stored  in the container:
    "CN=Schema,CN=Application Configuration,cn=Configuration".
    The built-in VAs are not cached and cannot be searched. You can access and modify these attributes using the Get, GetEx, Put, PutEx methods of the IADs interface. The built-in VAs for some object classes are described in Active Roles Schema.

  • Custom virtual attributes(CVA): These attributes can be created using the Active Roles snap-in. The attribute values are not stored in the Active Roles configuration database. The CVAs are calculated using values of the designated AD attributes of the directory object in question. You can also have Active Roles store the attribute values in the external databases, such as the HR database of your organization. The objects that represent the CVAs (objects of the attributeSchema type) are stored in the container:
    "CN=Virtual Attributes,CN=Server Configuration, cn=Configuration".
    CVAs are not cached and cannot be searched. To get/set the CVA values, you should apply the special policy scripts (see Applying Custom Virtual Attributes).
    Note    : You can retrieve the distinguished name for the container that stores the custom VAs using the edsvaVirtualAttributesDN property of the rootDSE object. For details, refer to "Binding to Active Roles-specific objects" in Related Topics.

  • Stored custom virtual attributes (SCVA): These attributes can be created using the Active Roles snap-in. Unlike the custom virtual attributes, the values of SCVAs are stored in the Active Roles configuration database. You can access and modify the SCVAs using the IADs interface or using the Advanced Properties dialog box for the object in question. To get/set the SCVA values, you don't need apply the special policy scripts. Note that use of the SCVAs can result in a considerable increase in the Active Roles database size.

Creation of Virtual Attributes

The procedure for creation of a new CVA or SCVA includes the following steps:

  1. Choose a name for the attribute.

  2. Determine the syntax of the attribute.

  3. Decide whether the attribute needs to be a single-valued or multi-valued attribute.

  4. Determine the object classes to which you want the virtual attribute to apply.

  5. Decide whether the attribute will be CVA or SCVA.

The Active Roles snap-in provides the Add Virtual Attribute Wizard that allows you to easily create a new virtual attribute to be applied to the desired object classes.

To create a VA with Active Roles snap-in

  1. Choose the Advanced mode for Active Roles snap-in display:

    • On the View menu, click Mode, and then in the View Mode dialog box, select the Advanced Mode option and click OK.

  2. Start the Add Virtual Attribute Wizard:

    • In the console tree, expand the Configuration node, expand the Server Configuration node, right-click Virtual Attributes, point to New, and then click Virtual Attribute.

  3. In the Attribute Identification window, do the following:

    • In the Common-Name and LDAP Display Name text boxes, specify the common name and the LDAP display name, respectively, for a new attribute.

    • In the Unique X.500 object ID text box, optionally, change the default value for the attributeID property (OID) for the new attribute. The wizard prompts for the OID value built upon base OIDs generated for Active Roles. If you want to generate your own value, you may use Microsoft's Oidgen tool (oidgen.exe).

    • In the Schema ID GUID text box, optionally, change the default value for the schemaIDGUID property. The wizard prompts for the schemaIDGUID value randomly generated by Active Roles. If you want the new attribute to have the fixed schemaIDGUID property, replace the default value with your own value. For example, you can generate GUID with the Microsoft Platform SDK's Uuidgen (uuidgen.exe) tool.

    When finished, click Next.

  4. In the Attribute Syntax window, do the following:

    • In the Syntax list, click the syntax to which you want the virtual attribute to conform.

    • Select the Multi-valued check box to make the virtual attribute multi-valued. Otherwise, leave this check box cleared.

    When finished, click Next.

  5. In the Object Classes window, under Select object classes, select check boxes next to the object classes to which you want the virtual attribute to apply. To display all available object classes, select the Show all possible classes check box. When finished, click Next.

  6. In the Attribute Storage window, select the Store values of this attribute in the Active Roles Administration database check box if you want to create a new SCVA. Otherwise, leave this check box cleared.

  7. Click Finish to complete the wizard.

Once you have created a VA, you can view or modify its properties with the Active Roles snap-in in the following way:

  1. In the console tree, expand the Configuration node, expand the Server Configuration node, and then click Virtual Attributes.

  2. In the details pane, right-click the VA in question, and then click Properties.

  3. On the General, Syntax and Classes tabs of the Properties dialog box, view or modify the desired properties of the VA.

Applying Custom Virtual Attributes

To process values of the custom VAs with methods provided by the IADs interface, you should develop and apply a policy script that handles the attribute value. The policy script may include the onPostGet, onPreModify or onPreSearch event handlers that allow you to manipulate the CVA value when the Active Roles client sends the request to get, modify or search for the VA, respectively.

To apply the CVA, perform the following steps:

  1. Create the CVA.

  2. Develop the appropriate policy script.

  3. Apply the policy script (follow the procedure in Applying Policy Scripts).

Reading Virtual Attributes Programmatically

To read ordinary attributes from a script, you normally use the Get method, for instance Get("description"). Although this operator is not applicable to virtual attributes.  To read virtual attributes, you must use the GetInfoEx method instead.

Using Virtual Attributes in Searches

You can safely use in searches the built-in virtual attributes which are marked as searchable in Active Roles Schema. Also, any custom virtual attributes marked as stored can be used in searches. The custom non-stored virtual attributes are not searchable and cannot be used in searches.

Sample Policy Script

The policy script below enforces the following administrative policy:
When the Active Roles client sends the request to modify the custom VA edsvaUserMaritalStatus, the script retrieves the value of that attribute from the object Request, and then stores it in the AD attribute extensionAttribute2. When the Active Roles client sends the request to retrieve the custom VA edsvaUserMaritalStatus, the policy script reads the value of the AD attribute extensionAttribute2, and then sets the property edsvaUserMaritalStatus in the object Request to the retrieved value.
For more information about the object Request, refer to Getting In-Process Property Values and Setting In-Process Property Values.

Note:

The attribute extensionAttribute2 is added to the AD schema only if you have installed Microsoft Exchange Server. Otherwise, you should use any other available attribute to store the value of the VA edsvaUserMaritalStatus. 

Sub onPostGet(Request)
 Dim buf
  ' Retrieve the processed object class
 If Request.Class <> "user" Then Exit Sub
  ' Verify whether the VA edsvaUserMaritalStatus is requested
 If Request.IsAttributeRequested("edsvaUserMaritalStatus") Then
    strUserDN=Request.ADsPath
     ' Bind to the processed user object
    Set objUser = GetObject(strUserDN)
     ' Retrieve the value of the attribute extensionAttribute2
     ' and set the attribute edsvaUserMaritalStatus in the object Request to that value
    objUser.GetInfoEx Array("extensionAttribute2"),0
    buf=objUser.Get("extensionAttribute2")
    Request.Put "edsvaUserMaritalStatus", buf
 End If
End Sub
Sub onPreModify(Request)
 Dim buf
  ' Retrieve the processed object class
 If Request.Class <> "user" Then Exit Sub
    buf=Request.Get("edsvaUserMaritalStatus")
  ' Verify whether the VA edsvaUserMaritalStatus is requested
 If buf <> vbEmpty Then
    strUserDN=Request.ADsPath
     ' Bind to the processed user object
    Set objUser = GetObject(strUserDN)
     ' Set the attribute extensionAttribute2 to the value retrieved from
     ' the request to modify the VA edsvaUserMaritalStatus
    objUser.Put "extensionAttribute2", buf
    objUser.SetInfo
 End If
End Sub

Sample Custom Scripts

The following custom scripts illustrate how you can set and get values of custom VAs for the user Blair, Joseph residing in the OU Users in the domain mycompany.com.

The following script sets the value of the attribute edsvaUserMaritalStatus to "Married".

' Set the user DN
strUserDN="EDMS://CN=Blair\, Joseph,CN=Users,DC=mycompany,DC=com"
Set usr=GetObject(strUserDN)
 ' Load the VA in question to the cache and set its value
usr.GetInfoEx Array("edsvaUserMaritalStatus"),0
usr.Put "edsvaUserMaritalStatus","Married"
usr.SetInfo

The following script gets the value of the attribute edsvaUserMaritalStatus.

' Set the user DN
strUserDN="EDMS://CN=Blair\, Joseph,CN=Users,DC=mycompany,DC=com"
Set usr=GetObject(strUserDN)
 ' Load the VA in question to the cache and get its value
usr.GetInfoEx Array("edsvaUserMaritalStatus"),0
on error resume next
strStatus=usr.Get("edsvaUserMaritalStatus")
MsgBox "User's marital status: " + strStatus
on error goto 0

Important:

To use these custom scripts, you should first create the custom VA (LDAP display name: edsvaUserMaritalStatus; single-valued; syntax: CaseIgnoreString; applies to: Users ), create the Policy Object that enforces the above script policy, and then link that Policy Object to the OU Users.
