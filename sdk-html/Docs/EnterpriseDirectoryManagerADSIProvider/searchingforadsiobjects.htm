Searching for Directory Objects Using ADO

Search operations enable you to find Active Directory and Active Roles-specific objects (referred to as directory objects) based on selection criteria (query) and to retrieve specified properties for the objects found. In comparison to Enumerating Directory Objects, the search operations are more powerful and advanced tools designed for searching for directory objects. This topic briefly discusses searching for directory objects with ActiveX Data Objects (ADO) that can be used in Automation applications (Visual Basic, VBScript, Active Server Pages, and so on).

To perform the search for directory objects with ADO

  1. Load the OLE DB provider using the Connection object.

  2. Specify a search string (search criteria) and additional search parameters using the Command object.

  3. Execute the search and process the search results using the RecordSet object.

  This topic details the steps of the procedure above.

Loading the OLE DB Provider

The Connection object allows you to load the OLE DB provider, and validates the user credentials. For details, see Connecting to ADSI Provider with ADO.

Specifying Search Criteria and Additional Search Parameters

In your application, you can specify the search criteria as a string containing parameters of the search. The search parameters include the scope of the search, the search filter, the attributes to retrieve, and the depth of the search.

Note:

If a search string includes the specific computer name, you should set the ADSI Flag property of the Connection object to 32768. For details, see Connecting to ADSI Provider with ADO and Sample Script below in this topic.

To specify your search parameters, you can use an LDAP search string or a SQL search string.

LDAP Search String

An LDAP search string consists of four parts separated with semicolons (;):

  • Scope of the search - specifies a container object, such as a domain or Organizational Unit, where the search will start. For example, the string <EDMS://ServiceCompName/OU=Sales,DC=gamp,DC=com> specifies that the search starts in the OU Sales. This string ensures that the provider will connect to the Administration Service running on the ServiceCompName computer. If the search scope does not include the specific computer name, the provider will connect to the Administration Service running on the local computer.
    The Active Roles ADSI Provider allows you to perform an attribute search query that enables a search of an object's distinguished name valued attributes (referred to as ASQ search). For information about the format of the search scope part to be used for ASQ search, refer to Searching Using an Attribute Scope Query.

  • Search filter - specifies the objects to be found according to the object classes, names or properties. For example, the filter (&(objectCategory=person)(objectClass=user)(!cn=andy)) allows you to find all users except the user with the property cn (common name) equal to andy.
    For more information about LDAP search filters, refer to LDAP Search Filter Syntax.

  • The object properties to be returned - a list of the found object's properties you want to return, such as ADsPath, department or manager.

  • Depth of the search - specifies limits of the search. You can determine three type of the search: a Base query limits the search to only the base object, a OneLevel query is restricted to the immediate children of the base object, but excludes the base object itself, a subTree query (or a deep search) includes all the objects beneath the base object, excluding the base object itself.

The following is an LDAP search string that founds only the user objects residing in the OU Sales and returns only values of the cn and description properties. 

"<EDMS://ServiceCompName/OU=Sales,DC=mydomain,DC=com>;(&(objectClass=user)(objectCategory=person));cn,description;OneLevel"

SQL Search String

A SQL search string has the following format:

SELECT <select-list> FROM <ADsPath> WHERE <search-condition> ORDER BY <sort-list>

The following table describes the keywords of the SQL query statement.

Keyword

Description

SELECT

<select list> specifies a comma-separated list of attributes to retrieve for each object. If you specify *, the query retrieves only the ADsPath of each object.

FROM

<ADsPath> specifies the ADsPath of the base of the search. For example, the ADsPath of the Users container in an Active Directory domain might be 'EDMS://CN=Users,DC=domain,DC=com'. The path is enclosed in single quotes ' '. Note that <ADsPath> can include the specific computer name (see description of the search scope for an LDAP search string).

WHERE

An optional keyword that specifies the query filter. For example, you can use as <search-condition> objectClass = 'user' AND sn='N*' to find only the users whose last names (sn property) begin with N.

ORDER BY

An optional keyword that generates a server-side sort. <sort-list> is a comma-separated list of the object properties on which to sort. You can use the optional ASC and DESC keywords to specify ascending or descending sort order; the default is ascending. The ORDER BY keyword overrides any sort key specified with the "Sort on" property of the ADO Command object. For example, you can use as <sort-list> sn.

Notes:

  • If you use the SQL syntax, the depth of the search can be specified by assigning a value to the searchscope property of the Command object as follows: Command.Properties("searchscope") = <value>, where <value> stands for a constant from the ADS_SCOPEENUM enumeration (ADS_SCOPE_BASE = 0, ADS_SCOPE_ONELEVEL = 1, or ADS_SCOPE_SUBTREE = 2). The default is ADS_SCOPE_SUBTREE.

  • Active Roles does not support the search by effective permissions when using the SQL or LDAP search string. 

Starting the Search and Processing the Search Results

To start the search, you should invoke the Execute method of the Command object. Invoking this method, you obtain the RecordSet object. The primary function of the RecordSet object is to enumerate the result set and obtain the data. The result set can contain values for properties that have single or multiple values. Getting a single-value attribute is straightforward, similar to getting the column value in the relational database (for example, Fields('name').Value). Getting an attribute with multiple values, however, is more challenging. In this case, the Field.Value is an array and you must check the lower and upper bound of the array, as illustrated in the following code snippet (VB language):

Set Com = CreateObject("ADODB.Command")
' Start the search process
Set rs = Com.Execute
For i = 0 To rs.Fields.Count - 1
    Debug.Print rs.Fields(i).Name, rs.Fields(i).Type
Next i
'--------------------------
'Navigate the record set.
'--------------------------
rs.MoveFirst
lstResult.Clear 'Clear the user interface.
While Not rs.EOF
    For i = 0 To rs.Fields.Count - 1
        'For Multi Value attribute
        If rs.Fields(i).Type = adVariant And Not (IsNull(rs.Fields(i).Value)) Then
            Debug.Print rs.Fields(i).Name, " = "
            For j = LBound(rs.Fields(i).Value) To UBound(rs.Fields(i).Value)
                Debug.Print rs.Fields(i).Value(j), " # "
                lstResult.AddItem rs.Fields(i).Value(j)
            Next j
        Else
           'For Single Value attribute.
           Debug.Print rs.Fields(i).Name, " = ", rs.Fields(i).Value
           lstResult.AddItem rs.Fields(i).Value
        End If
    Next i
    rs.MoveNext
Wend

Sample Script (VBScript)

The following script shows how to find the user accounts with the property "department" beginning with "Sales" and change the property value to "Sales Department (London)". The script uses the SQL search string.

Dim oConnect
Dim Command
Dim rs
' Create the Connection object and load the OLE DB provider
Set oConnect = CreateObject("ADODB.Connection")
oConnect.Provider = "ADsDSOObject"
' Set the ADSI Flag property to 32768
oConnect.Properties("ADSI Flag") = 32768
' Validate user credentials ("", "" means that the current user credentials are used)
oConnect.Open "DS Query", "", ""
' Create the Command object
Set Command = CreateObject("ADODB.Command")
' Maintain an active connection
Set Command.ActiveConnection = oConnect
' Specify the search string (SQL syntax)
Command.CommandText = _
  "SELECT ADsPath FROM 'EDMS://OfficeComputer/OU=London,DC=mycompany,DC=com' WHERE objectClass = 'user' AND objectCategory = 'person' AND department = 'Sales*'"
' Set additional properties of the search string
Command.Properties("Page Size") = 2
Command.Properties("Timeout") = 30 'seconds
Command.Properties("searchscope") = 1 'ADS_SCOPE_ONELEVEL
' Start the search
Set rs = Command.Execute
Dim usr,n
n=0
' Process the search results
Do Until rs.EOF
    ' Bind to the found user object using its ADsPath property
    Set usr = GetObject(rs.Fields("ADsPath").Value)
    ' Change the department property value
    usr.Put "department", "Sales Department (London)"
    ' Send changes you have made to Administration Service
    usr.SetInfo
    n=n+1
    rs.MoveNext
Loop
MsgBox "The department property of " & CStr(n) & " user account(s) has been changed"

Note:

The Active Roles ADSI Provider uses the read-only OLE DB provider. This means that you cannot issue the UPDATE statement in the SQL statement.
To modify properties of a found object, you should get its ADsPath property, bind to that object using the GetObject function, and then modify the desired properties using the Put or PutEx method of IADs.
