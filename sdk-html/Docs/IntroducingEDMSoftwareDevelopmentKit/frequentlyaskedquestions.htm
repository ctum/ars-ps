Frequently asked questions

Can I use scripts to manage Computer Resources with Active Roles?

Yes. In this documentation, you can find some examples of scripts for Computer Resources management (see Managing Computer Resources).

What development languages can be used to write policy scripts?

Any scripting language supported by the Microsoft Windows Script Host, such as Windows PowerShell, VBScript and JScript, can be used to develop policy scripts.

What development languages can be used to work with Active Roles ADSI Provider?

Any language that supports COM objects can be used to work with the Active Roles ADSI Provider, such as C, C++ and C#, as well as automation-compliant languages such as Microsoft Visual Basic and Microsoft Visual Basic Scripting Edition (VBScript).

Can policy scripts display messages to Active Roles users?

Policy scripts can only return error messages by generating run-time errors. If a pre-event handler generates an error, the error message is displayed to the Active Roles user and Administration Service terminates the execution of the in-process operation. If a post-event handler generates an error, the error message is recorded into the EDM Server event log without terminating the in-process operation. For more information about using custom error messages in policy scripts, see Returning Custom Error Messages.

Can policy scripts prompt Active Roles users for data?

Currently, Active Roles does not enable policy scripts to prompt Active Roles users for data.

In what security context do policy scripts run?

Policy scripts are executed in the security context of the user account under which the Administration Service service is running.

Do policy scripts run synchronously or asynchronously?

Whether a policy script runs synchronously or asynchronously depends on the type of the event handler that receives control. The following event handlers are executed as part of the requested administrative operation:

  • Any pre-event handler

  • onGetEffectivePolicy

  • onCheckPropertyValues

  • onPostGet

When such a handler receives control, the service continues to process the request only after the script execution completes.
The situation changes with post-event handlers (except onPostGet): the service executes post-event handlers independently of processing the requested operation. For example, a post-delete handler may continue to run after the deletion operation is completed.

When the Active Roles user defines a script policy, is the policy script automatically replicated among Administration Services?

Since Administration Service stores policy scripts in Administration Database, the scripts are replicated as part of the overall Active Roles configuration information among all Administration Services that are configured to be replication partners.

Policy script fails to retrieve a property value from the in-process object although the property has a value in Active Directory. What's wrong?

The in-process object (object Request) only contains the property values that are required to complete the requested operation. For example, if the client requests the service to modify the Description property but not the other properties of a user object, only the new value for the user Description appears in the object Request. Since the object Request contains no information about the other properties, attempts to retrieve their values will fail.

How can a policy script access a property value that is not in the object Request?

The only property values in the object Request are those required to complete the requested operation. To retrieve the value of a property that is not in the object Request, a policy script needs to use the ADSI provider. For example, the following code (VBScript) retrieves the Department property:

Set objUser = GetObject(Request.Get("ADsPath"))
objUser.GetInfoEx Array("department"), 0
vPropertyValue = objUser.Get("department")

For more examples, see Binding to the Object Being Processed, DirObj object.

How can a policy script determine if a property value is set in the object Request?

All the property values that are defined in the object Request (in-process property values) can be enumerated and retrieved using methods of the interface IADsPropertyList.

Can policy scripts keep user-defined variables in the objects Request?

Yes. The Parameter property of the object Request makes it possible to get and set the values of user-defined variables by variable name. For more information, refer to IEDSRequestParameters.

It seems that the onPostMove event handler does not work. What's wrong?

In a script policy applied to a container, the onPostMove event handler is activated after an object is moved into that container, and is not activated after an object is moved from that container. To clarify this behavior, suppose the script policy PA is applied to the container A, while the script policy PB is applied to the container B. When there arrives a request to move an object from A to B, the service first invokes the onPreMove event handler in PA, then performs the requested action, and finally, after the object has been moved from A to B, the service invokes the onPostMove event handler in PB, rather than in PA. This behavior is because the object now resides in the container B and, therefore, should be affected by the policy just applied to B. See also Event Handlers.

Is it possible to control the order in which policies are processed?

Active Roles provides the ability to control the order in which policy entries are processed within a Policy Object. You can use the Up and Down buttons on the Policy Settings tab in the Policy Object Properties dialog box to adjust the execution order of the policy entries. The higher an entry stands in the list, the earlier it is executed. Currently, Active Roles provides no ability to control the order in which individual Policy Objects are processed.

I have a script that works with the Microsoft ADSI (LDAP) Provider. How should I modify the script for use with the Active Roles ADSI Provider?

You should replace LDAP: with EDMS: in all binding strings used in your script. For example, the string
LDAP://CN=TestUser,DC=DEV,DC=FABRIKAM,DC=COM
should be replaced with
EDMS://CN=TestUser,DC=DEV,DC=FABRIKAM,DC=COM
