Customizing Active Roles

Active Roles provides the opportunity to customize its off-the-shelf functionality through creating scripts and applications interacting with Administration Service. It allows a high degree of customer modification to meet specific business and organizational needs, giving customers superior flexibility in how to use the product, and enabling them to build solutions that can be easily integrated with existing systems and data. The list below provides some ways the product can be customized:

  • Using the Active Roles ADSI Provider, the existing proprietary applications could communicate with Active Roles to perform administration and provisioning tasks on user accounts and groups

  • With the Active Roles ADSI Provider, a custom web-based interface could be developed for users to change their own passwords, enforcing specific corporate rules for the password complexity

  • Using policy scripts, custom corporate rules could be enforced to regulate data format and administrative workflows

  • With policy scripts, the data stored in a Human Resources database or ERP system could be incorporated into the process of user accounts administration and provisioning 

Active Roles makes it possible for user-developed scripts and applications to manipulate directory objects through Administration Service (persistent objects) as well as take control of objects that are in the process of being created, modified, or deleted with Active Roles (in-process objects).

Having a means of programmatic access to persistent and in-process objects on Administration Service makes it easy for developers to customize Active Roles in these two areas: creating custom applications and user interfaces, and implementing the enforcement of corporate administrative policies based on running custom scripts (so-called script policies.

Creating Custom Applications and User Interfaces

A custom application or user interface can be created and used to manipulate directory objects through Active Roles. Active Roles offers the developer an efficient means to communicate with Administration Service - the Active Roles ADSI Provider. The ADSI Provider enables custom applications and scripts to access directory objects using interfaces that conform to the Microsoft ADSI specification.

Custom applications are usually executables that provide data to Administration Service or retrieve and process data from Administration Service. For example, an organization with a separate Human Resources database could develop and deploy a custom application that provides personal information from the database to Administration Service in order to facilitate the process of user account provisioning.

Custom user interfaces are usually web-based interfaces that help distribute certain tasks to users. Custom user interfaces can also be used to streamline the workflow of network administrators and help-desk operators. For example, web-based pages could be created so that help-desk operators are only shown the fields related to the user properties help desk is permitted to view and modify according to the corporate standards.

Both custom applications and user interfaces rely on the ADSI Provider to gain access to the functionality of Active Roles. For more information, see Active Roles ADSI Provider.

Implementing Custom Script Policies

Providing the ability to implement administrative policies based on running user-developed scripts (custom script policies), the Active Roles ADSI Provider makes it easy for policy scripts to manipulate both in-process and persistent object data so as to:

  • Facilitate user account provisioning - populate user properties through external database integration and automate multi-step provisioning tasks

  • Ensure directory content integrity - prevent inconsistency of Active Directory data by enforcing update sequence and data format policies across the enterprise

  • Enforce business rules - maintain security design and capture administration expertise by integrating business rules into the administrative workflow

Once configured, custom script policies are enforced without user interaction. Active Roles automatically takes care of executing the policy scripts that supplement particular administrative operations and trigger additional administrative actions. For example, policy scripts can be used to:

  • Perform a sophisticated validity check of the input data

  • Synchronously change information in multiple data sources, such as the Active Directory store, Exchange server, and HR or ERP-system database

  • Ensure that the Active Roles users follow a prescribed administrative workflow

  • Link multiple administrative tasks into one operator transaction

For more information, see Script Policy and Scheduled Tasks.
