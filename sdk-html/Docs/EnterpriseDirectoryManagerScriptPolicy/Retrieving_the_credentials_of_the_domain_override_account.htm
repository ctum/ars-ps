DomainOverrideCredentials Object

When you design your policy script or workflow activity script, you may need the credentials of the user account that Active Roles uses to access a given domain. For example, you may want to access data or resources that require credentials of a user account with sufficient rights in that domain. As the policy and workflow activity scripts are normally run under the Active Roles service account, which may not have sufficient rights in the desired domain, the credentials of a different user account are required to perform certain commands under that user account. To address this requirement, Active Roles provides the ability to retrieve the credentials of the override account that is specified in the domain registration information stored by Active Roles.

For each domain registered with Active Roles, you have the option to supply the user name and password of a user account under which you want Active Roles to access the domain. This user account is referred to as an override account. If no override account is specified in the domain registration information, then Active Roles accesses the domain with the Active Roles service account. For a domain that has an override account specified, it is possible to obtain the credentials of the override account from the $DomainOverrideCredentials object in a policy script or workflow activity script. This object is available only in Windows PowerShell scripts.

$DomainOverrideCredentials.Get method

For each managed domain that Active Roles accesses using an override account (the user account specified in the domain registration information, to be used for access to the domain instead of the Active Roles service account), the $DomainOverrideCredentials object holds the credentials of the override account. You can use the Get method of this object to obtain a PSCredential object containing the credentials of the override account for a given domain, and then pass the PSCredential object to PowerShell cmdlets to gain access to data and resources in that domain. You can also use the $DomainOverrideCredentials object to obtain the override account credentials for a managed AD LDS instance. In this case, the argument of the Get method should be the DN of any object in a directory partition on that instance.

Syntax

$DomainOverrideCredentials.Get(DNstring)

Arguments

DNString

[in] Required. String. Identifies the Distinguished Name (DN) of the domain whose override account credentials you want to retrieve. To identify the domain, you can supply the DN of any object from the domain naming context (for example, the DN of a user or group) instead of the DN of that domain itself.

Return value

PSCredential object containing the credentials of the override account. If the domain does not have an override account specified, no value is returned.

Example

The following example demonstrates how to create a folder on a network file share for the user account being created by Active Roles, with the folder creation performed by accessing the file server under the override account of the domain. The name of the folder is set to the name of the user account. The variables in this example:

$path:  UNC path to the network file share on the file server where you want to create the folder.

$userName:  Name of the user account.

$userDN:  Distinguished Name of the user account.

$credentials:  PSCredential object containing the credentials of the override account.

Once these variables have been populated, the New-Item cmdlet is used to create the folder.

Note that this example requires an override account to be specified for the domain in question; otherwise, the $credentials variable holds nothing.

 

Function onPostCreate($Request)

{

    $path = "\\fileServer.domain.com\fileShare"

    $userName = $Request.Get("cn")

    $userDN = $Request.DN

    $credentials = $DomainOverrideCredentials.Get($userDN)

 

    $scriptBlock = { New-Item -Path $args[0] -ItemType directory -Name $args[1] }

    Start-Job -ScriptBlock $scriptBlock -Credential $credentials -ArgumentList @($path, $userName)

}
