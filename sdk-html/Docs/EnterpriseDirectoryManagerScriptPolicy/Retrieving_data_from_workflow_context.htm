Retrieving data from workflow context

When creating a script to be used in a workflow activity, you may need your script to retrieve and use information from the workflow run-time context, such as:

  • The value of a certain workflow parameter (parameter value)

  • The object returned by a certain Search activity (found object)

  • The object created or modified by a certain workflow activity (activity target)

  • The value of a certain attribute saved by a Save Object Properties activity (saved object property)

  • The name of the button that the approver chose to click when performing the approval task (approver action choice)

This topic provides information and examples demonstrating how to retrieve:

  • Workflow name

  • Parameter value

  • Found object

  • Activity target

  • Saved object property

  • Approver action choice

Active Roles has a built-in object named $workflow that provides the properties and methods for retrieving this workflow context data. The $workflow object is accessible from Windows PowerShell scripts held in Active Roles script modules and executed by Active Roles workflow activities or script policies.

Retrieving workflow name

To identify the workflow that runs a given script, you can use the WorkflowName property of the $workflow object in that script.

WorkflowName property

Use the WorkflowName property to retrieve the name of the workflow that the $workflow object represents.

Syntax

$workflow.WorkflowName

Return value

This property returns the name of the workflow definition object from which the current workflow instance is derived.

Retrieving parameter value

Workflow parameters and their values are normally defined at the workflow design time. Some parameters may be configured so that the user can supply or change the parameter value when starting the workflow by hand. In either case, the workflow parameter values are determined upon starting the workflow, stored in the workflow run-time environment, and can be retrieved by using the Parameter or ParameterEx method of the $workflow object.

Parameter method

Use the Parameter method to retrieve the value of a single-value parameter.

Syntax

$workflow.Parameter(name)

Arguments

name

[in] Required. String. Identifies the name of the parameter whose value you want to retrieve.

Return value

Depending upon the syntax of the parameter, this method returns the String, DateTime, ObjectGUID, DN, SID, or SecureString object representing the parameter value.

ParameterEx method

Use the ParameterEx method to retrieve the values of a multi-value parameter.

Syntax

$workflow.ParameterEx(name)

Arguments

name

[in] Required. String. Identifies the name of the parameter whose values you want to retrieve.

Return value

Depending upon the syntax of the parameter, this method returns an array of the String, DateTime, ObjectGUID, DN, SID, or SecureString objects representing the parameter values.

Example

The following script function is intended to be used by a Script activity included in an automation workflow. This function starts Windows service "QSVC" on the remote computers specified by workflow parameter "Computers". To perform this task, the function logs on to each remote computer with the user name and password specified by the "LogonName" parameter and "Password" parameter, respectively.

The workflow parameters used by this script must be configured as follows:

  • Computers - Multi-valued, String parameter

  • LogonName - Single-valued, String parameter

  • Password - Single-valued, SecureString parameter

For instructions on how to create a workflow parameter, see the Active Roles Administrator Guide.

Function StartService()

{

    $computers = $workflow.ParameterEx("Computers")

    $logonName = $workflow.Parameter("LogonName")

    $password = $workflow.Parameter("Password")

    $securePassword = ConvertTo-SecureString $Security.Cryptography.decryptfromstring($password) -asplaintext -force

    $cred = New-Object System.Management.Automation.PSCredential ($logonName, $securePassword)

 

    foreach ($computer in $computers)

    {

        trap {continue}

        $ps = New-PSSession -computername $computer -Credential $cred

 

        Invoke-Command -session $ps -scriptblock { Start-Service QCSVC }

        Remove-PSSession $ps

    }

}

Retrieving found object

Found objects represent the results of a search performed by a certain Search activity. You can retrieve them by using the FoundObject method of the $workflow object.

FoundObject method

Use the FoundObject method to retrieve the object returned by a certain Search activity.

Syntax

$workflow.FoundObject(name)

Arguments

name

[in] Required. String. Identifies the name of the Search activity whose search results you want to retrieve.

Return value

An object possessing the methods and properties of the IADs and IADsPropertyList interfaces with the exception of the GetInfo, GetInfoEx, SetInfo, ResetPropertyCache methods.

Example

The following script function disables the user accounts found by the Search activity named "Find Inactive Users".

Function DisableInactiveUsers($Request)

{

    disable-QADUser $workflow.FoundObject("Find Inactive Users").get("distinguishedName")

}

Retrieving activity target

The target object of a given activity is the directory object the activity acts upon. For a create activity, this is the object to be created; for a change or delete activity, this is the object to be modified or deleted, respectively. You can retrieve the target object of the activity by using the ActivityTarget method of the $workflow object.

ActivityTarget method

Use the ActivityTarget method to retrieve the target object of a certain activity. For example, you can use this method to retrieve the object created by a "Create" activity.

Syntax

$workflow.ActivityTarget(name)

Arguments

name

[in] Required. String. Identifies the name of the activity whose target object you want to retrieve.

Return value

An object possessing the methods and properties of the IADs and IADsPropertyList interfaces with the exception of the GetInfo, GetInfoEx, SetInfo, ResetPropertyCache methods. The property cache of the object returned by this method is initialized with the following attributes of the corresponding directory object: distinguishedName, displayName, objectClass, objectSid, objectGUID, sAMAccountName.

Example

The following script function disables the user account created by the "Create" activity named "Create Shadow Account".

Function DisableShadowAccount($Request)

{

    disable-QADUser $workflow.ActivityTarget("Create Shadow Account").get("distinguishedName")

}

Retrieving saved object property

Active Roles workflow provides the Save Object Properties activity type for saving properties of a particular object at workflow execution time. The properties are saved in the workflow data context, and can be retrieved by other activities before or after the object has changed. In a workflow that includes an activity of the Save Object Properties type, you can use the SavedObjectProperties method to retrieve object properties saved by that activity.

SavedObjectProperties method

Syntax

$workflow.SavedObjectProperties(activityName).get(attributeName)

$workflow.SavedObjectProperties(activityName).getEx(attributeName)

Arguments

activityName

[in] Required. String. Identifies the name of the Save Object Properties activity.

You should specify the display name of the Save Object Properties activity that exists in the workflow; otherwise, the method returns no value.

attributeName

[in] Required. String. Identifies the LDAP display name of the attribute you want to retrieve.

You should specify the name of one of the attributes listed in the "Target properties" setting of the Save Object Properties activity; otherwise, the method returns no value.

Return value

The attribute value or values as of the time of executing the Save Object Properties activity.

$workflow.SavedObjectProperties(activityName).get(attributeName) returns a single value in case of a single-value attribute or an array of values in case of a multi-value attribute.

$workflow.SavedObjectProperties(activityName).getEx(attributeName) returns an array of values for a single-value or multi-value attribute. In case of a single-value attribute, the array contains a single element.

Example

In a workflow that starts upon deletion of an object and uses a Save Object Properties activity to save the object's properties before the object is deleted, you can use the following script function to retrieve the object's display name after the object is deleted. This example assumes the name of the activity that saves object properties in the workflow is "Save Object Properties 1".

Function SavedDisplayName($Request)

{

    return $workflow.SavedObjectProperties("Save Object Properties 1").get("displayName")

}

Another example demonstrates how to process the values of a saved multi-value attribute:

Function onPostDelete($Request)

{

    $OtherHomephones = $workflow.SavedObjectProperties("Save Object Properties 1").GetEx("otherHomePhone")

    foreach($homeNum in $OtherHomephones)

    {

        #

        # Your code goes here that uses the retrieved $homeNum

        #

    }

}

Retrieving approver action choice

When performing an approval task, the approver is required to choose one of the action buttons provided by the approval rule. Action buttons appear on the pages for performing the approval task. Each button applies a certain action to the task. Normally, two built-in buttons, named "Approve" and "Reject" by default, are displayed for each approval task. The approval rule can be configured to add custom action buttons. Depending on the button's action type, clicking a custom action button causes the workflow to allow ("Complete" action type) or cancel ("Reject" action type) the operation that is subject to approval. You can use the ApproverAction method of the $workflow object to retrieve the name of the action button the approver chose to click, and then have your script perform the custom actions appropriate to the button chosen by the approver.

ApproverAction method

Use the ApproverAction method to retrieve the name of the action button the approver chose to click on the pages for performing the approval task.

Syntax

$workflow.ApproverAction(name)

Arguments

name

[in] Required. String. Identifies the name of the Approval activity that created the approval task.

Return value

The name of the action button the approver chose to click.

Example

The following script function checks to see if the approver has clicked the "Cancel Operation" action button when performing the approval task, and, if so, updates the virtual attributes edsaCustomActionActivator and edsvaCustomActionDescription accordingly. In this example, "Custom Approval" is assumed to be the name of the Approval activity whose approval tasks you want the script to keep track of.

Function CheckApproverActon($Request)

{

    if ($workflow.ApproverActon("Custom Approval") -eq "Cancel Operation")

    {

        $request.ResetPropertyItem("edsaCustomActionActivator")

        $request.Put("edsvaCustomActionDescription", "Operation cancelled")

    }

}
