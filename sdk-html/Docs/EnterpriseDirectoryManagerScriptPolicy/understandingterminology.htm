Understanding Terminology

This topic defines some basic terms used in the Active Roles script policy, scheduled tasks, approval rules, and entitlement profile.

Script Policy and Scheduled Tasks Related Terms

Administrative Policy

The term administrative policy refers to a set of rules that constrain the interaction that an Active Roles user wants to have with an object. These may be requirements to only accept data changes that meet certain format conditions (property validation policy), running scripts that supplement administrative actions (script policy), or creating additional objects, such as user mailboxes and home folders, when Active Roles creates directory objects.

Policy Scripts

Script policy is based on running custom scripts referred to as policy scripts. The policy scripts are stored in the Active Roles configuration database and replicated to all Administration Services configured as replication partners. Each policy script includes one or more event handlers, which are procedures Administration Service activates upon client requests to perform administrative actions such as the creation, modification, or deletion of objects. Using policy scripts makes it easy to define a wide range of constrains, trigger additional actions, and combine a number of administrative tasks into a single batch. When executed by the Administration Service, those scripts can use the objects exposed by the Service, such as DirObj, EventLog, and Cryptography.

Events and Event Handlers

In the script policy, events refer to actions that occur in response to particular administrative operations requested by Active Roles clients. Event handlers are functions within a policy script that have a predefined syntax and serve as procedures to be activated by Administration Service whenever particular events occur. Event handlers divide into two categories: "pre-event" handlers and "post-event" handlers. Administration Service triggers "pre-event" handlers before it commits the requested changes, and triggers "post-event" handlers immediately after the changes are successfully committed.

Scheduled Task

The term scheduled task refers to a script the Administration Service executes at scheduled time. The scheduled task scripts are stored in the Active Roles configuration database and replicated to all Administration Services configured as replication partners. When executed by the Administration Service, those scripts can use the objects exposed by the Service, such as Task and EventLog.

Library Scripts

The term library script refers to a script that contains functions (Sub or Function procedures, as applied to VBScript) available for other scripts. For example, a function saved into a library script can be called by several policy or scheduled task scripts. This makes your source codes more compact.

Script Modules

Active Roles stores policy scripts, library scripts and scheduled task scripts in the Active Roles configuration Database as script modules. You can develop script modules using any scripting language supported by the Microsoft Windows Script Host, such as Windows PowerShell, VBScript and JScript.

For policy scripts, each module defines one or no handler for each event. When configuring a script policy, you first select a script module to add it into the Policy Object, and then select events upon which the script will be activated. If the script does not specify event handlers for some of the events you have selected, the script takes no effect when those events occur.

When configuring a scheduled task, you first specify the schedule, and then select a script module you want the Administration Service to run.

Approval Activity Related Terms

Approval Rule

Ensemble of settings created by the Administrator to configure approval activity in a workflow. Approval rules are used to specify the operations that are subject to approval, as well as who is authorized to complete approval tasks.

Operation Requiring Approval

According to the approval rules conditions, some changes attempted in Active Directory (such as object creation, deprovisioning, deletion, etc.) may require approval to be performed. If this is the case, the approval rule creates an entity called operation, which becomes a part of the approval activity and serves as the basis for approval tasks generation.

Approval Conditions

The criteria that an approval rule uses to determine whether or not an operation is subject to approval. If these conditions are met the approval activity starts.

Initiator

A user that initiated an approval-dependant operation.

Approvers

Users or groups of users designated to complete approval tasks by approving or rejecting operations. One or more Approvers are assigned by Administrator when configuring an approval rule.

 

Note

For more information, see "Approval Activity" the in Active Roles Administrator Guide.
