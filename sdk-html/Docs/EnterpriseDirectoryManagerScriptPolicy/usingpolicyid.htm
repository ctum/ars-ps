Using Policy Identifier

The Active Roles Administration Service automatically assigns a unique identifier (policy ID) to each Policy Object Entry (referred to as policy) you create with Active Roles snap-in or using Policy Object interfaces. Custom policies (such as script policies, mailbox creation policies, etc.) are marked with binary policy IDs while Active Roles built-in policies have fixed policy IDs listed in this table.

Active Roles allows you to assign a user-defined policy ID to your script policies with the special Function procedure (VBScript) onGetPolicyMarker. This function must return the policy ID (any string) you want to assign to your script policy.  You should insert onGetPolicyMarker in the source of your policy script.
For example, the following code snippet assigns the policy ID "MyPolicyMarker" to your script policy:

Function OnGetPolicyMarker()
    OnGetPolicyMarker = "MyPolicyMarker"
End Function

The use of onGetPolicyMarker enables you to completely override a policy. You can also override only the specified event handler in your script policy.

Overriding Policies

You can override both the Active Roles built-in policies and your custom script policies.

If two policies have the same policy ID, Administration Service will consider them as the same policy. In this case, Administration Service will process the policy that is linked closer to the directory object the policy affects.

To clarify this, consider an example: you have created two Policy Objects (PO1 and PO2) and each PO includes a script policy marked with the same policy ID. PO1 is linked to an Organizational Unit (OU) while PO2 is linked to a user object residing within that OU. In this case, the user object will be affected only by PO2 while other directory objects that reside in that OU will be affected only by PO1.

If you have marked your script policy with one from the Active Roles built-in policy identifiers, your script policy will override the corresponding Active Roles built-in policy. For more information, please refer to Overriding Active Roles Built-in Policies.

Overriding Event Handlers

You can override the specified event handler(s) of the same type defined in your script policies. Event handlers of other types will be not affected.

To override an event handler

  1. Using onGetPolicyMarker, assign any policy ID to the script policy (SP1) that includes the event handler you want to override.

  2. In other policy script (SP2), within the event handler of the same type, call the method OverridePolicy (provided by the interface IEDsRequestParameters) and pass that policy ID to this method.

To clarify this procedure, consider the following example.

These code snippets (SP1 and SP2) illustrate how you can disable the onPreModify event handler in SP1. Administration Service will activate the onPreModify event handler defined in the script SP2 if you send a request to modify a User object. Otherwise, the event handler defined in SP1 will be activated.

SP1 

[VBScript]
Sub onPreModify(Request)
' Here goes your code
'
End Sub
' Identify the Policy
Function onGetPolicyMarker()
 onGetPolicyMarker="Script1"
End Function
[PowerShell]
function onPreModify($Request)
{
    # Here goes your code
    #
}
# Identify the Policy
function onGetPolicyMarker()
{
    return "Script1"
} 

SP2 

[VBScript]
Sub onPreModify(Request)
' Here goes your code
'
' Disable the onPreModify event handler in the policy script SP1
If Request.Class = "user" Then
    Request.OverridePolicy "Script1"
End If
End Sub




[PowerShell]
function onPreModify($Request)
{
    # Here goes your code
    #
    # Disable the onPreModify event handler in the policy script SP1
    If($Request.Class -eq "user")
    {
        $Request.OverridePolicy("Script1")
    }
}

Note:

In this example, the script SP2 is thought to be processed earlier than SP1. Otherwise, the use of the OverridePolicy method takes no effect.
