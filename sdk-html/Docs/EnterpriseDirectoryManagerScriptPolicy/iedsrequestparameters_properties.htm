IEDSRequestParameters Properties

The following table describes properties provided by the IEDSRequestParameters interface.

Note:

The LdapFilter, FilterSearchResults and SearchPref properties are only accessible from the onPreSearch event handler.

Property

Description

FilterSearchResults

Access: Write-only.

Data type: Boolean.

The default value is FALSE. 

When this property is set to TRUE, the onPreSearch event handler activates the handler onFilter. The handler onFilter is a boolean function you should add to your policy script source code. onFilter is called each time the search adds a new record (a found object) to the search result set. When onFilter returns FALSE, the corresponding record is removed from the search result set.

The use of the FilterSearchResults property and the handler onFilter allows you to filter the search results for certain Active Roles clients. To clarify this, consider the sample policy script below in this topic.

LdapFilter

Access: Read/Write

Data type: String

Sets and returns the Ldap search filter that the ADSI Provider uses to search for directory objects. The filter specifies the objects to be found according to the object classes, names or properties.

For more information about the Ldap search filters, please refer to Searching for Directory Objects.

Parameter

Access: Read/Write

Data type: String

Sets and returns values of the server-defined or user-defined variables in the in-process (request) object. The user-defined and server-defined variables are specific to the object Request and their lifespan is limited to the lifespan of the request.

The user-defined variables enable the interaction between different script modules, and between different event handlers within the same module. For example, you might define a certain variable during the execution of a pre-event handler and use that variable during the execution of a post-event handler. A user-defined variable is created by name when the script sets its value.

ObjectRequest.Parameter(Name)[=UserVarValue]

Arguments

ObjectRequest
Required. Always a name of the object Request.
Name
[in] Required. String value that identifies the name of the variable. For more information, see Server-defined and User-defined variables below in this topic.
UserVarValue
[in] Optional. When provided, assigns the specified value to the user-defined variable identified by the Name argument. 

For examples that illustrate the use of the Parameter property method, please refer to  Using Request Parameters.

SearchPref

Access: Read/Write

Data type:

Sets and returns the search preferences.

The search preferences are set by the Active Roles client that sent the request to search for directory objects. Using the SearchPref property, your policy script can read and modify the specified search preferences.

For more information about the search preferences, refer to the MSDN library.

ObjectRequest.SearchPref(Pref)[=PrefValue]

Arguments

ObjectRequest
Required. Always a name of the object Request.
Pref
[in] Required. Identifies the search preference in question you want to read or modify.
PrefValue
[in] Optional. When provided, assigns the specified value to the search preference identified by the Pref argument.

Settings

The argument Pref can take any value from the enumeration ADS_SEARCHPREF_ENUM. The syntax to which the argument PrefValue must conform depends on the corresponding preference.

Server-Defined and User-Defined Variables

Server-defined variables

The following table lists the names of server-defined variables.

Name

Description

Type

Identifies the type of the requested operation, such as the object creation, deletion, or modification. All possible values of this variable are listed later in this topic.

OriginalObject (deprecated)

In a request to copy an object, identifies the distinguished name of the object to be copied.

Note:

The OriginalObject variable is considered obsolete. While this variable is still supported in the current version of Active Roles, it may be removed in the future. When writing new scripts, you should avoid using this variable. When modifying existing scripts, you are strongly encouraged to remove any dependency on this variable. Use the EDS_CONTROL_SOURCE_OBJECT_DN Active Roles control instead.

TargetContainer

In a request to move an object, the value of this variable identifies the distinguished name of the destination container folder.

NewName

In a request to rename an object, the value of this variable identifies the new name in the form "cn=<name>" (except OUs) or "ou=<name>" (for OUs).

LdapServer

The value of this variable identifies the DNS name of a domain controller for the object's home domain. This variable is useful when you need to bind to the directory object involved in the requested operation.

DomainNC

The value of this variable identifies the distinguished name of the object's home domain.

NTMixedDomain

This variable is set to "True" when the object's home domain is in mixed functional mode; otherwise, this variable is set to "False".

RootDomainNC

The value of this variable identifies the root domain of the forest to which the object's home domain belongs.  

SelfRequest

This variable is set to "True" if the request was sent by the server or policy. Otherwise, it is set to "False".

RequestSource

The value of this variable identifies the request source. All possible values of this variable are listed later in this topic.

InterestingRequestType

The value of this variable identifies the type of the requested operation when you use the onGetEffectivePolicy or onCheckPropertyValues event handler. It takes the same values as the Type variable except EDST_REQ_GET_EFFECTIVE_POLICY and EDST_REQ_MAKE_COMPLIANCE. All possible values are listed later in this topic.

GcServer

Contains the DNS name of the global catalog server.

User-defined variables

You can create a user-defined variable by assigning a value to a variable with a name different from any server-defined variable name. For example, you can define the "MyVar" user-defined variable and assign it to the "myValue" value in the  following way.

[VBScript] 
Request.Parameter("MyVar")="myValue"




[PowerShell] 
$Request.Parameter("MyVar", "myValue")

Example Code

This policy script illustrates how to have Active Roles filter the search results for clients that are not Active Roles Admins. After you create a Policy Object that includes this script policy and link that Policy Object to the domain mycompany.com, the Active Roles clients that are not Active Roles Admins cannot see child objects of the OU OnlyforARAdmins.

[VBScript]
Sub onPreSearch(Request)
    Set objSession = Security.Sessions.Current
     ' Activate the handler onFilter if the client is not  Active Roles Admin
    If objSession.IsAdmin = False Then Request.FilterSearchResults = True        
End Sub
Function OnFilter(Request)
    OnFilter = True
     ' Hide children of the OU OnlyforAdmins
    If Request.Parent = "EDMS://OU=OnlyforARAdmins,DC=mycompany,DC=com" Then  OnFilter = False
End Function




[PowerShell]
function onPreSearch($Request)
{
  $objSession = $Security.Sessions.Current
  # Activate the handler onFilter if the client is not  Active Roles Admin
  if($objSession.IsAdmin -eq $false){$Request.FilterSearchResults = $true}
}
function OnFilter($Request)
{
  # Hide children of the OU OnlyforAdmins
  if($Request.Parent -eq "EDMS://OU=OnlyforARAdmins,DC=mycompany,DC=com")
  {
    return $false
  }
  else
  {
    return $true
  }
}

 

The following code snippet illustrates how you can change the Ldap search filter using the LdapFilter property. In this example, the policy does not allow you to perform the search operations on all users that belong to the Sales department. 

[VBScript]
Sub onPreSearch(Request)
    Dim Filter
    ' Read the Ldap search filter from the object Request
    Filter=Request.LdapFilter
    ' Modify the Ldap search filter
    Request.LdapFilter="(&" + Filter + "(!(&(objectClass=User)(department=Sales))))"
End Sub




[PowerShell]
function onPreSearch($Request)
{
      # Read the Ldap search filter from the object Request
   $Filter = $Request.LdapFilter
        # Modify the Ldap search filter
   $Request.LdapFilter = ("(&" + $Filter + "(!(&(objectClass=User)(department=Sales))))")
}
