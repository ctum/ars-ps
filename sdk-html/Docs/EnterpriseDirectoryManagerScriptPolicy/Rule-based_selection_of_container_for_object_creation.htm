Example: Rule-based selection of container for object creation

This topic demonstrates how to implement a scenario of creating user accounts in the Organizational Unit (OU) being automatically selected by Active Roles based on the value of the City (l) attribute of the user account to be created.

Suppose, the OU structure in your Active Directory is organized so that each office location has a separate OU. All user accounts for a given office location are stored in the OU assigned to that location. When creating a new user account, the helpdesk operator needs to select the appropriate OU and then run the New User command on that OU.

To simplify the process of creating new user accounts, you may want to implement a solution where helpdesk would only need to specify the office location and have the appropriate OU selected automatically. Such a solution should provide a certain "victim" OU for the New User command, a page on which the helpdesk operator chooses the office location, and a rule-based mechanism that uses the office location data to select the appropriate OU. The user creation process should be performed as if the New Use command is run on that office-location OU rather than the "victim" OU, so as to execute all policies and workflows that apply to the office-location OU. You can implement this solution by using the ChangeParentDN method in a script workflow activity to change the name of the "victim" OU to the name of the office-location OU in the user creation request.

The following example assumes domain.company.com to be an Active Directory domain containing these Organizational Units:

  • OU=London,DC=domain,DC=company,DC=com  The user accounts for the London office must be in the "London" OU.

  • OU=Paris,DC=domain,DC=company,DC=com  The user accounts for the Paris office must be in the "Paris" OU.

  • OU=New York City,DC=domain,DC=company,DC=com  The user accounts for the New York City office must be in the "New York City" OU.

  • OU=User Creation,DC=domain,DC=company,DC=com  Helpdesk will use the "User Creation" OU to run the New User command.

Our goal is to configure Active Roles so that the "New User" command on the "User Creation" OU in the Active Roles Web Interface prompts to select the office location (London, Paris, or New York City) and then creates the user account in the OU specific to the selected office location.

The configuration steps are as follows.

  1. Create a Property Generation and Validation policy for the "City (l)" user attribute that forces the attribute value to be selected from the list of the admissible office locations. Configure a provisioning policy object with the following policy rule and apply that policy object to the "User Creation" OU.

Policy rule: 'City' must be specified, and

      must be

             'London' (default value) or

             'Paris' or

             'New York City'

      Upon object creation, this policy generates default value: No

  1. Create a PowerShell script module containing the following policy script:

function onPreCreate($Request)

{

    changeCreationContainer($Request)

}

function onGetEffectivePolicy($Request)

{

    changeCreationContainer($Request)

}

function changeCreationContainer($Request)

{

    $city = $Request.Get("l")

    

    if($city -eq $null -or $city -eq [string]::Empty) { return }

    

    $ou = getContainerName($city)

    

    if($ou -eq $null -or $ou -eq [string]::Empty) { return }

    

    $Request.ChangeParentDN('OU=' + $ou + ',DC=domain,DC=company,DC=com')

}

function getContainerName($city)

{

    $city = $city.ToLower()

    switch($city)

        {

             'london' { 'London' }

             'paris' { 'Paris' }

             'new york city' { 'New York City' }

              default { $null }

        }

}

  1. Create a change workflow with the following start conditions:

Operation that starts this workflow: Create

Target object type: User

Initiator: Any user

Container: User Creation OU.

  1. Add a script activity to the workflow, above the "Operation execution" line. Configure the script activity to use the script you created in Step 2. Choose the onPreCreate function to run.

  2. On the workflow definition object in the Active Roles console, use the Advanced Properties command to set the edsaWorkflowPriority attribute to a value less than 500. This ensures that the workflow that chooses the OU where to create the user is executed prior to any other workflow that may also be matched to the operation of creating a user. For more information about the use of the edsaWorkflowPriority attribute, see "Workflow Processing Overview" in the Active Roles Administrator Guide.

  3. Customize the Web Interface by adding an entry for the "City (l)" attribute on the "General" tab of the form associated with the "New User" command on the menu for the "Organizational Unit (organizationalUnit)" object type.

  4. Configure Active Roles security so that the helpdesk operators have the "Create User" permission on only the "User Creation" OU, and are not allowed to create users in any other container.

Once you have completed these configuration steps, your helpdesk operators can create user accounts by using the "New User" command on the "User Creation" OU in the Web Interface. On the first step of the user creation wizard, the Active Roles Web Interface requires the operator to select the appropriate office location from a list; then, Active Roles ensures that the user account is created in the OU specific to the office location selected.
