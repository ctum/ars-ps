Understanding Library Scripts

In addition to policy scripts and scheduled task scripts, Active Roles defines library scripts. Library scripts are source codes that include functions or procedures to be used by policy scripts, scheduled task scripts, and approval rules.

When to Use Library Scripts

Use library scripts when you want to make your scripts more compact. For example, you can call a function saved into a library script from several policy or scheduled task scripts.

Storing Library Scripts

Active Roles stores library scripts in the Active Roles configuration database as script modules (objects of the edsScriptModule class). The script modules reside within the Script Modules configuration container. The container's DN: 

CN=Script Modules,cn=Configuration

For library script modules, the attribute edsaScriptType is set to 2.

Since library scripts are saved in the Active Roles configuration database, they will be available for all Administration Services participating in the Active Roles replication.

Creating Library Scripts

You can create library scripts in one of the following ways:

  • With the Active Roles snap-in: Use the New Object - Script Module wizard.

  • With your custom script: See the sample script below in this topic.

This following sample script creates a library script module and imports an existing script to that module. 

[VBScript] 
Set FSO = CreateObject("Scripting.FileSystemObject")
Set ScriptFile = FSO.OpenTextFile("ScriptSource.vbs", 1)
Set ContainerObject=GetObject("EDMS://CN=Script Modules,cn=Configuration")
Set newScriptModule=ContainerObject.Create("edsScriptModule", "CN=TestLibraryScriptModule")
newScriptModule.put "edsaScriptLanguage", "VBScript"
newScriptModule.put "edsaScriptText", ScriptFile.ReadAll
 ' Set the script module type: 2 - for library script
newScriptModule.put "edsaScriptType", 2
newScriptModule.SetInfo




[PowerShell]
$ContainerObject = [ADSI]"EDMS://CN=Script Modules,cn=Configuration"
$NewScriptModule =  $ContainerObject.Create("edsScriptModule", "CN=TestLibraryScriptModule")
$NewScriptModule.Put("edsaScriptLanguage", "VBScript")
$NewScriptModule.Put("edsaScriptText", [System.IO.File]::ReadAllText("C:\ScriptSource.vbs"))
$NewScriptModule.Put("edsaScriptType", 2)
$NewScriptModule.SetInfo()

The script creates the library script module TestLibraryScriptModule, and then import the VBScript saved in the file ScriptSource.vbs to that module. The script source can include any functions (Function or Sub procedures, as applied to VBScript) that will be used by other scripts.

Using Library Scripts

After you register a library script on Administration Service (see "Creating Library Scripts" earlier in this topic), any other script can employ functions or procedures the library script includes.

Note that the use of library scripts for VBScript differs from that for PowerShell.

VBScript

For VBScript, to have any script use procedures included in a library script, perform the following steps:

  1. In the source code of the script that will use a library script, define an object that represents the library script:
    Set <library script> = ScriptLib.Load("Script Modules/<library script>")

    The <library script> refers to the name of the library script you want to use.

  2. To call a procedure the library script includes, in the script source, insert the following line:
    <library script>.<procedure name> <comma-separated list of the procedure arguments, if any>

    The <procedure name> refers to the name of the procedure you want to call. The library script must contain this procedure.

To clarify the steps above, consider the following example.

Let us register the MyLibraryScript library script module on Administration Service. The library script includes the Parameters procedure that determines the object class and AdsPath to the in-process object.
The MyLibraryScript library script module contains the following code:

[VBScript]
Sub Parameters(ByVal Request, ByRef ObjClass, ByRef ObjAdsPath)
     ObjClass = Request.Class
     ObjAdsPath = Request.AdsPath
 End Sub
Let us register the following policy script on Administration Service: 
[VBScript]
Dim MyLibraryScript, strClass, strPath
Set MyLibraryScript = ScriptLib.Load("Script Modules/MyLibraryScript")
Sub onPreCreate(Request)
    MyLibraryScript.Parameters Request, strClass, strPath
 '
 ' Your code goes here
 '
End Sub

This policy script calls the Parameters Sub stored in the library script MyLibraryScript. The variables strClass and strPath return the object class and AdsPath of the in-process object, respectively.

PowerShell

For Windows PowerShell, to have any script use procedures included in a library script, perform the following steps:

  1. In the source code of the script that will use a library script, define the onInit event handler that contains the following line:
    $context.UseLibraryScript("<library script>")
    The <library script> refers to the name of the library script you want to use.

  2. To call a procedure the library script includes, in the script source, insert the following line:
    <procedure name>     (<comma-separated list of the procedure arguments, if any>)
    The <procedure name> refers to the name of the procedure. The library script must contain this procedure.

To clarify the steps above, consider the following example.

Registering the MyLibraryScript library script module with the Administration Service. The library scripts includes the Parameters function that determines the object class and AdsPath to the in-process object.
The MyLibraryScript library script module contains the following code:

[PowerShell]
function Parameters($Request, [ref] $ObjClass, [ref] $ObjAdsPath)
  {
     $ObjClass = [string]$Request.Class
     $ObjAdsPath = [string]$Request.AdsPath
  }
Registering the following policy script with the Administration Service: 
[PowerShell]
function onInit($context)
  {
    $context.UseLibraryScript("MyLibraryScript")
  }




function onPreCreate($Request)
  {
    Parameters($Request, $strClass, $strAdsPath)
    #
    # --- Your code goes here ---
    #
  }

This policy script calls the Parameters function stored in the MyLibraryScript library script. The variables strClass and strAdsPath return the object class and AdsPath of the in-process object, respectively.
